# 05 — Коллаборация разных LLM-моделей

**Тема каталога:** #1
**Кластер A-J:** A
**Приоритет:** Б (9)
**Уровень готовности:** ★★★

## Что это

**Multi-LLM collaboration** — паттерн, при котором несколько разных моделей работают над одной задачей: одна генерирует, другая критикует, третья проверяет факты, четвёртая редактирует.

Это не «ансамбль» в ML-смысле, а **разделение ролей** между моделями разных провайдеров (Claude, GPT-5, Gemini, Mistral, локальные).

## Зачем это нужно

1. **Качество** — модели компенсируют слабости друг друга
2. **Специализация** — каждая модель в роли, в которой она сильна
3. **Стоимость** — дешёвые модели для рутины, дорогие для критичных шагов
4. **Resilience** — нет single point of failure
5. **Приватность** — часть шагов на локальной модели

## Паттерны коллаборации

| Паттерн | Пример |
|---------|--------|
| Generator + Critic | GPT-5 пишет → Claude проверяет |
| Specialist + Generalist | Code-Llama пишет код → Claude рецензирует |
| Cheap-then-expensive | Llama 3.1 черновик → Claude финал |
| Voting | 3 модели → majority answer |
| Debate | 2 модели спорят → 3-я выносит вердикт |
| Pipeline | NER → классификатор → генератор |

## Применение в проектах автора

1. **SozialPlaner** — Llama 3.1 (черновик Widerspruch) → Claude (правовая проверка) → DeepL (перевод DE/RU)
2. **Newsroom AI** — GPT-5 (новость) → Claude (фактчек) → Llama (генерация заголовков)
3. **CareMate** — локальная модель (мед-данные) → Claude (только обезличенные сводки)

## Реализация

- **LangGraph** — граф из узлов-моделей с условными переходами
- **CrewAI** — агенты с ролями
- **AutoGen** (Microsoft) — диалоговые сценарии между агентами
- **DSPy** — оптимизация цепочек с разными моделями

## Открытые вопросы

- Балансировка стоимости / качества
- Обнаружение «согласованных ошибок» (когда обе модели ошибаются одинаково)
- Cache-стратегии (одни и те же запросы к разным моделям)
